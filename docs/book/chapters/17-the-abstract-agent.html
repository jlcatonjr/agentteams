Chapter 17: The Abstract Agent

Part IV: Emergence and Abstraction

1. Convergent Design Across Independent Codebases

When independent codebases face structurally similar constraints, they converge on similar architectural patterns. Miller and Drexler (1988a), in their work on computational ecologies, observed that open systems composed of interacting computational agents develop recurring structural regularities not because a designer imposes them but because the environment selects for them. The same selective logic operates in agent-team architecture. Multiple repositories in a workspace, developed at different times and for different purposes, may each grow their own agent populations. The populations range from roughly ten to twenty agents. Despite their independence, they share patterns: functional agents separated from domain agents, invariant cores protecting constitutional rules, extension points permitting local adaptation, authority hierarchies resolving conflicts among information sources, living document policies preventing stale accumulation, security enforcement gates before destructive operations.

Although the architecture of each repository agent team was created by myself, there is also parallel development, not sequential imitation, in which each architecture evolved in corresponding ways. At some point, I analyzed and systematized the patterns. This is akin to the pattern Menger (1883) identified as the hallmark of institutional emergence, in which common structures arise from action of actors responding under similar, and often related, constraints rather than from deliberate design. In several cases the same pattern (the invariant core block, for instance) appeared in multiple repositories before being systematized. The patterns share logic without sharing specifics: one repository's pipeline stages differ from another's abstract governance structure, yet both implement staged knowledge processing with local producers passing outputs through defined channels. This correspondence is given systematic ordering via abstract agents.

And the convergence across independent repositories suggests that the constraints of the problem space select for this architecture. Varied interactions with similar constraints produce similar solutions, and the common structure is visible only after the fact. The patterns that survive this selection are products of structured interactions. The environment selects for configurations that generate order because configurations that do not generate it accumulate governance failures that degrade the team’s ability to function. Provision of self-correcting feedback loops, for example, is required. Either they will be provided by repeated human intervention, they will be automated, or project progress will be stalled because neither path is pursued.

2. The Abstract Agent Layer as Platform

Once I observed the convergence, the common patterns were extracted into a designated workspace-level repository. This repository houses what I call the abstract agent layer: governance agents that define patterns applicable to any repository in the workspace. The extraction is a standard software engineering operation (pulling common abstractions upward from concrete implementations) but its significance here is primarily architectural.

The abstract layer is a living document that provides a general structure that governs the evolution of all agents in conversation with it. The daily pipeline contributes to and learns from individual repositories. In Chapter 1's terms, the abstract layer is infrastructure capital (§1.10) whose governance templates provide the sources of truth from which agents across repositories instantiate and maintain specialists (§1.2). It includes abstract versions of the functional agents (orchestrator, cleanup, security, and others) with invariant cores that all repositories must preserve. These cores define the common communication architecture discussed in Chapter 2 and the plan-structure from Chapter 3 that makes each repository's resources complementary at the ecosystem level (across repositories, not only within its own project).

I extracted the abstract layer bottom-up from observed convergence, not top-down as a governance framework. When I noticed that independently developed repositories all had similar orchestrator structures, security agents, and cleanup workflows, the common patterns were pulled upward into the workspace layer. The abstraction followed the convergence, codifying what practice had already settled on.

But codification introduces a tension. Once extracted, the abstract layer takes on a life as a platform: a canonical configuration source that individual repositories inherit from and must conform to. The platform constrains the repositories that adopt it, even as those repositories' innovations may feed back into the platform. This tension between bottom-up convergence and top-down enforcement, between configuration inheritance and local adaptation, is the central architectural problem of the remaining chapters. Chapter 22 will return to it.

2.1 The Template System as Abstract Layer

The abstract layer takes a concrete organizational form in the agentteams module: a library of reusable templates organized across three tiers. The templates/universal/ directory contains the ten functional governance templates — orchestrator, navigator, security, conflict-auditor, conflict-resolution, code-hygiene, adversarial, cleanup, agent-updater, and agent-refactorer. The templates/domain/ directory contains sixteen production templates covering content composition, quality auditing, cohesion repair, style enforcement, format conversion, figure generation, compilation, and reference management. The templates/builder/ directory contains the team-builder agents that conduct an intake interview and invoke the rendering engine to instantiate a complete agent team. The three-tier structure is the extraction described above: common patterns pulled from independent practice and made portable.

The universal/domain split encodes the functional/domain bifurcation of Chapter 3. Functional agents have a consistent structural profile across any project inheriting from the abstract layer: the same tool declarations, the same constitutional-rule sections, the same handoff chains to security and auditing agents. Production agents vary across projects. The domain templates accommodate this variation by capturing each production function without binding it to any project's output. A functional template gives the governance agent the coordination framework it requires. A domain template gives the production agent the brief-intake and quality-routing structure it requires. The structural difference between the two template types — in YAML tool declarations, authority-source references, and handoff targets — is the architectural expression of the governance/production distinction in a form the rendering engine can process.

Two constitutional documents define the template system's own governance. The authoring guide specifies what every template must contain: required YAML frontmatter fields, required body sections in fixed order, and tier-specific content requirements. The placeholder-conventions document defines the syntax for variable content at instantiation time. They occupy the same role for the template library that the orchestrator's invariant core occupies for each project: the rules the platform enforces on itself.

3. Bidirectional Propagation

The daily pipeline's Stage 8 pushes configuration best-practices from the workspace-level documentation layer to individual repositories. This is top-down configuration propagation. The canonical source distributes its governance templates to all inheriting instances.

Separately, Stages 10 and 11 handle abstract pattern propagation: Stage 10 discovers emergent patterns across repositories through the agent-investigator (Chapter 18), and Stage 11 distributes remediation instructions to each repository's local orchestrator. Stage 8 propagates what the developer has already codified. By contrast, Stages 10 and 11 propagate what the system has discovered through its own investigation. In both cases, the domain knowledge remains local. A repository's domain agents, chapter outlines, and project-specific workflows are untouched by propagation. Only the governance substrate travels.

The more interesting direction is bottom-up. The agent-abstraction-reviewer (Section 4), in the course of its compliance checking, generates reports that flow back to the abstract layer. When it encounters a pattern in a repository that is not yet represented in the abstract layer, it flags it as a candidate for abstraction. These flagged patterns are innovations that the abstract layer did not anticipate: instances where a repository solved a problem in a way that the platform has not yet codified. Each novel-pattern report is a compressed signal (a knowledge proxy in Chapter 2's sense) transmitting a repository's innovation upward so that the abstract layer can evaluate whether the common structure belongs in the platform's canonical definitions.

The bottom-up flow is how the abstract layer grows. Growth comes from recognizing and codifying patterns that emerge in practice across repositories, not from top-down decree. The two directions together create a feedback loop: abstract patterns are tested in practice through top-down propagation, and practice generates new patterns through bottom-up synthesis. Without the variety produced by independent repositories developing their own solutions, the abstract layer would be static, a snapshot of the developer's initial design rather than an evolving platform that incorporates the lessons of practice.

4. The Meta-Agents of the Abstract Layer

The abstract layer is maintained by three meta-agents, each responsible for a distinct function. The orchestrator (at the workspace level) coordinates the overall propagation workflow. The collector-discovery agent scans the workspace to identify which repositories have agent teams and should receive propagation. The agent-abstraction-reviewer compares repository-level agent files against their abstract templates, classifying differences.

The three-agent architecture is a separation of concerns applied to configuration management. The orchestrator knows workflow procedures but not repository topology. The collector-discovery agent knows where agent teams exist but not what their templates should look like. The reviewer knows template specifications but not the broader workflow context. No single meta-agent holds all of the propagation knowledge. Miller and Drexler (1988c) described computational ecologies in which specialized agents interact through defined protocols, each contributing its local expertise to a system-level outcome that none could produce alone. The propagation pipeline instantiates this ecology: the orchestrator handles sequencing, the collector handles topology, the reviewer handles compliance, and the protocol that connects them (stage reports, classification schemas, propagation manifests) is the medium through which their local knowledge combines.

A fourth meta-agent exists in the abstract layer. The agent-investigator stands apart from the propagation workflow. Its role is bottom-up discovery, complementing the reviewer's top-down compliance checking. The investigator is the subject of Chapter 18.

5. The Invariant Core / Extension Point Architecture

Abstract patterns face a design problem. They must be universal enough to apply across all repositories but flexible enough to accommodate the particular needs of each project. A pattern that is too rigid will be violated or worked around. One that is too flexible offers no coordination value.

The solution is the template method pattern applied to governance configuration. An abstract agent file specifies two things. The invariant core contains the authority hierarchy, security rules, and workflow constraints that apply everywhere. The clearly marked extension points contain slots for domain routing, project-specific workflows, and custom strategies. When propagated to a repository, the invariant core is preserved word-for-word. The extension points are filled with project-specific content. The abstraction reviewer (Section 4) verifies that invariant cores remain intact across all repositories.

Invariant cores are constitutional constraints (Chapter 4): rules that repositories cannot override without violating the ecosystem's governance contract. Extension points are the degrees of freedom those constraints leave open, the project-specific arrangements within the universal plan-structure (Chapter 3) that make each repository uniquely productive.

A project-level orchestrator illustrates the architecture. It inherits the abstract orchestrator's invariant core verbatim: responsibility definitions, security requirements, constitutional rules. But it adds extension points for project-specific workflows, domain agent routing to production and governance specialists, and the specific authority hierarchy that places the principal's published research above project documentation. The core ensures consistent governance. The extensions enable the production of a particular work.

5.1 Placeholder-Driven Extension Points

The invariant-core/extension-point architecture takes operational form through two categories of placeholder tokens. The rendering engine populates auto-resolved placeholders, written {UPPER_SNAKE_CASE}, from the project manifest at instantiation time. It derives values such as {PROJECT_NAME}, {PRIMARY_OUTPUT_DIR}, and {AUTHORITY_HIERARCHY} automatically from the project description the developer supplies. Manual-required placeholders, written {MANUAL:UPPER_SNAKE_CASE}, require operator-supplied values the engine cannot infer: specific file paths, external repository locations, tool API surfaces, style-guide references. The rendering engine collects these tokens into a setup checklist at generation time, making explicit what the operator must provide before the team is complete.

The two categories implement the invariant/extension distinction at the token level. Auto-resolved tokens are the abstract layer's fixed interface: they define what the platform requires every project to supply in standardized form. Manual-required tokens are the extension points: the binding locations where project-specific knowledge enters the template and turns it into a particular team rather than an abstract pattern. An agent file with literal {MANUAL:} tokens still present at deployment is an incomplete instantiation. Detecting unreplaced {MANUAL:} tokens is an initialization-completeness test, not a documentation-maintenance task.

6. Classification of Differences

When the agent-abstraction-reviewer compares a repository's agent file against its abstract template, not every difference has the same significance. The reviewer classifies each difference into one of three categories. An intentional extension is project-specific content placed in a designated extension point. This is expected and desirable. It is the whole purpose of the extension-point architecture. An accidental drift is a divergence in the invariant core from the abstract template. This is a defect to be corrected, the kind of drift analyzed in Chapter 21. A novel pattern is something the repository has developed that the abstract template does not yet represent. This is a discovery to be evaluated.

The three-class taxonomy is a form of static analysis applied to configuration files. In conventional static analysis, a tool inspects source code against a rule set and classifies deviations: errors (violations of the language spec), warnings (likely bugs), and informational notes (style suggestions). The reviewer's classification serves the same function at the architectural level. Drift is an error, a violation of the governance contract. Intentional extensions are expected, conforming uses of the designated flexibility. Novel patterns are informational, signals that the rule set itself may need updating.

Novel patterns (the third category) are the generative cases. They represent innovations that individual repositories have embedded in their local practice, innovations that might be valuable to the entire ecosystem if abstracted. The reviewer flags them for human evaluation. If adopted, they join the abstract layer's repertoire, and the next propagation cycle carries them to all repositories.

Butos and McQuade (2005) analyzed adaptive classifying systems: systems that do not sort inputs into fixed categories but revise their categories in response to the inputs they encounter. The reviewer's classification is adaptive in this sense. The three-class framework is stable, but the content of the "novel pattern" class feeds back into the abstract layer, potentially generating new invariant cores or extension points that reshape the classification landscape for future reviews. The reviewer does not just classify. Its classifications change what there is to classify.

Once the common patterns have been extracted into a platform and the propagation machinery is in place, the question shifts. How does the system discover what the platform does not yet contain? The abstraction reviewer catches deviations from known patterns. The investigator, the subject of Chapter 18, scans for patterns the system has not yet named.

7. Team Generation as Codified Abstraction

The abstract agent layer generalizes governance patterns within an existing ecosystem of repositories. But generalization is not generation. Knowing the common patterns of agent teams does not, by itself, create a new team for a project that has none. The agentteams Python module occupies this second function: it takes the architectural knowledge crystallized in the abstract layer's templates and schemas and uses it to generate complete, coordinated agent teams from scratch.

The module accepts a project-description JSON file as its input contract. The project description specifies the project's name, purpose, output artifacts, primary information sources, authority structure, and the technical context within which agents will operate. This input is validated against a schema that encodes the book's taxonomy: the four-tier architecture of orchestrator, governance, domain, and workstream-expert agents is embedded in the schema's structural requirements. A project description that violates the schema cannot proceed. The schema is not merely a data validation mechanism. It is an executable statement of the architectural theory this book develops.

From the validated project description, the module constructs a team manifest: a structured specification of which agents to instantiate, with what parameters, in what tier of the hierarchy. The manifest is the abstract layer's templates made concrete for this project. An ingest → analyze → render → emit pipeline then populates each template, resolving auto-resolvable placeholders from the project manifest and surfacing manual-required ones for operator completion. The output is a complete collection of agent files, one per role, ready for deployment to the target framework.

The relationship between the abstract layer and the agentteams module is that of platform and factory. The abstract layer is the living governance infrastructure described in Sections 2 through 6: a platform of templates, bidirectional with individual repositories, evolving as patterns abstract upward and standards propagate downward. The agentteams module is a codified factory built on top of that platform. Given a new project's description, it reads the canonical templates, instantiates them for the project's particulars, and emits a complete agent team. The factory does not generate arbitrary teams. It generates teams whose architecture embodies the platform's current canonical state, which is itself the codification of the convergent practice that Section 1 described.

This is the recursive structure that the rest of the book's argument describes only partially. Convergent practice across repositories generates patterns. Patterns abstract upward into the platform. The platform's codification generates the factory. The factory generates new teams from new project descriptions. Those teams, as they develop, produce novel patterns that bottom-up synthesis (Section 3) returns to the abstract layer. Theory and practice are not sequential here: the theory generates the tool, the tool generates the practice, and the practice revises the theory.
