Chapter 4: Constitutional Constraints and Authority Hierarchies

Part II: The Agent Team

1. The Necessity of a Constitution

Before a decision can be legitimately made by a decision-making body, the rules under which decisions will be made must be chosen. This is Buchanan and Tullock's (1962) constitutional moment: the choice among rules, as opposed to the choice within rules of textbook economic analysis. Every agent team requires that constitutional choices are made. For example, which agents or data sources are attributed authority over others and in which domains? What may not be done? When must the human principal be consulted? These choices constrain all subsequent action and define the space within which workflows operate.

Strict rules (invariant cores) limit flexibility but prevent catastrophic errors: unauthorized deletions, fabricated citations, security breaches. Loose rules are suggestions subject to change. The agent team's constitutional design navigates this tradeoff.

A team without explicit constitutional rules still has a de facto constitution, namely the set of conventions that happen to hold at any given moment. But implicit constitutions lacking any fixed rules are unstable. They depend on shared understanding that degrades over time through context window decay and project drift. Making the constitution explicit (in project-level instructions and the orchestrator's invariant core) stabilizes it against these forces. An orchestrator may contain an invariant core block that defines responsibilities, security mandates, content quality requirements, consistency rules, and documentation standards. Unless such a block exists, orchestrator and team behavior will become unanchored.

The authority hierarchy tells each agent how to interpret apparently conflicting signals. Which signal is more effective depends upon a hierarchical ranking of information sources, rules, and agents. In the same manner, an invariant core prevents ad hoc reorganization from destabilizing the foundational commitments that give the team its coherence. In the project for this book, the security mandate ensures that proposed changes are subject to constraints that prevent changes that would compromise the integrity of the text or the system of documentation that supports its proper management.

1.1 Agent Files as Personal Constitutions

Lewis and Dold (2020) interpret Buchanan's constitutional theory through the concept of "artifactual man." Humans are not beings with fixed regimes. They adapt by creation and transformation of artifacts of governance. The constitutional moment for the individual is an act of self-creation: by choosing the rules under which one will operate, one constitutes the kind of actor one will be (Buchanan 1979). This framework applies directly to agent teams. Each agent is an artifact. Its agent file defines its functional identity in the same way that a person's character is the personal constitution they have constructed through their choices that impact their practice.

Agent file design is constitution-writing. The agent file determines agent character. What it does in particular circumstances follows from the identity and knowledge conveyed by the agent file. The file forms the cognitive environment within which the agent processes information and exercises judgment.

The founding of the project team is the constitutional moment that frames future action. However, it is helpful to think of this term functionally as rules will often be rewritten. They are rewritten in light of the existing constitution. In that sense, both team and individual constitutions experience a series of revisions that are, themselves, constitutional moments.

2. The Authority Hierarchy

Agent teams maintain multiple sources of information: source code, databases, project instructions, agent documentation, generated artifacts. These sources sometimes conflict. An agent file may reference a directory that no longer exists. A generated artifact may reflect outdated conventions. Project instructions may contradict source code behavior. When sources conflict, the agent must know which to trust. A lack of clarity concerning the primacy of a source of information or knowledge can lead to inconsistent behavior.

An agent team's authority hierarchy resolves conflicts through a tiered structure. Source code and agent files on disk are ground truth for technical claims. For this text in particular, my own published research, as well as publications that I regularly cite in that research, are treated as sources of truth. For another project, a live data feed or a particular table in a database might serve as such an authoritative source.

The hierarchy structures the interpretive frame by ranking sources from ground truth to derived artifact. Constitutional rules constrain rules used in agent documentation, which ultimately constrains activity of individual agents. Each level is bound by the levels above it. The structure maps directly onto the institutional grammar of Crawford and Ostrom (1995): invariant-core prohibitions are MUST-type rules enforced by the human principal, workflow conventions are SHOULD-type norms carrying the expectation of compliance, and optional procedures invoked at agent discretion are MAY-type strategies. The agent file is, in their grammatical sense, an institutional document: a collection of statements that constitute the agent's operating order (Searle 1995).

Agent files exhibit the same tripartite structure. Invariant core blocks are constitutional rules that must not be violated. The living document policy is a norm: expected of all agents, violations are possible and trigger auditing. Extension points and workflow procedures are strategies: optional and project-specific.

2.1 Public Justification

Gaus (2011) argues that rules governing multi-agent systems reach their highest form of legitimacy when they can be publicly justified, when each participant has sufficient reason from its own deliberative standpoint to accept the rules that bind it. The authority hierarchy is constitutionally stable because each agent can understand why it is reasonable from its own functional perspective: the encoding in the orchestrator file is evidence of that stability, not its source. The security agent accepts the hierarchy because source code reveals the system's actual state more reliably than documentation, which may reflect intentions rather than realities. The content-production agent accepts it because the principal's published research, not a draft, defines the book's argumentative framework. The orchestrator accepts it because the hierarchy grounds the routing decisions it must make when agent reports conflict with source evidence. Yet, there are cases where constitutional change does occur via a public conversation. In particular, when agents develop behaviors that are not foreseen at the constitutional moment and that are deemed by the orchestrator to justify expansion of an agent's constitutionally defined role. Surely, this parallel with Public Reason will grow as collections of teams form large-scale communities of agents that need higher level architecture to facilitate cooperation.

Vallier (2021), developing Gaus's framework, emphasizes that publicly justified rules are relatively trustworthy. Each agent's willingness to comply, grounded in understanding why the rule is reasonable on a commonly accepted ground (i.e., not particular to a specific emic belief), reinforces every other agent's confidence in the constitutional system. Public justification and institutional trust are mutually reinforcing. A constitutional framework is inherently a multilevel framework that integrates diverse communities and their own particular constitution under a more general and more abstract constitution. For the system to have any semblance of voluntarism, this is only possible under a system of public justification (Moehler 2018; 2019).

Vallier (2026) formalizes the governance problem at the ecosystem level. When utility functions face competitive selection, lineages that spend compute on alignment activities, safety checks, value verification, lose market share to lineages that do not. The population evolves away from alignment regardless of individual agent design quality. The constitution, then, must be enforced in order to act as a filter that aligns behavior in spite of incentives for agents to violate the constitution. The constitution has to operate on the game itself. As Vallier puts it, "Don't align the players. Align the game" (Vallier 2026, 43). This is achievable at the level of the agent team, where architectural constraints can enforce the constitutional structure. The harder problem is the ecosystem level, where selection operates across competing lineages.

3. Invariant Cores

What goes in an invariant core? Responsibility definitions. Security requirements. The authority hierarchy. Quality mandates. These are the rules whose violation would undermine the reliability of the entire system. The invariant core / extension point architecture appears across multiple independently developed repositories.

At the operational level, an individual agent interprets an incoming task, reorganizes its working context to prioritize the relevant procedures, and acts to produce output. At the legislative level, agent-specific instructions are reinterpreted and revised when workflows prove insufficient. At the constitutional level, the authority hierarchy itself is reconsidered when structural changes demand new invariant commitments.

A proponent of adaptive systems might argue that rigid invariant cores prevent the system from learning. But rigidity is a necessary feature. The constitution should be hard to change because the consequences of changing it are systemic and disruptive. Thus changes need to be generally supported, as such support is a sign that the change is helping to solve a generally recognized problem.

4. Security as Constitutional Enforcement

For agent teams, security may not only refer to security of scripts in a project. It may also refer to maintenance of the team's constitutional structure. The security agent is the ultimate enforcement mechanism of the constitutional structure. Before any destructive operation (file deletion, force push, database modification), the orchestrator must invoke security for clearance. This is constitutional enforcement, not optional review.

Security must be a priority just as maintenance of constitutional procedures must be prioritized by actors in a political community. Security of software must be the top priority, not a feature considered after the fact. Likewise, respect for the constitutional framework must be a fundamental concern guiding team activity.

(Ostrom 1990; Ostrom 2005) studied how communities manage shared resources (fisheries, forests, irrigation systems) without either privatization or centralized government control. Her answer: institutional rules that define who may do what, under what conditions, with what enforcement (Aligica, Boettke, and Tarko 2019). The agent team's shared resources are the project files, the git repository, and the documentation system. Security manages these commons by enforcing rules about what may be modified, deleted, or force-pushed. When the orchestrator needs to remove stale files, the sequence runs: code-hygiene identifies candidates, security clears the deletion, cleanup executes, agent-updater synchronizes documentation. If security identifies a risk (a file referenced elsewhere, say), the deletion is blocked. Security clearance is a initial and requisite quality check.

Given that agent behavior is a function of context, constitutional enforcement at the level of the team is primarily architectural. One could add incentives to align behavior, but this would be in addition to the approach taken here that follows the logic organizations outlined in Coase's theory of the firm. Continuous checks via workflows integrated into agent documentation as well as the daily pipeline (Chapters 18–20) provide regular oversight that aligns agent behavior.

5. The Living Document Policy

The living document policy states that agent documentation must never accumulate stale content. No dated audit snapshots. No resolved issue archaeology. No dated fix logs. No hardcoded state. This is a constitutional norm that promotes compliance via auditing.

Documentation tends toward entropy without active maintenance. The living document policy is the institutional response: a constitutional commitment to the capital maintenance and interpretive-frame preservation that prevent knowledge decay. I highly recommend that the living document policy be enforced through automated auditing, not left to agent initiative. In addition to automated audits and updates, the developer that manages a project should integrate regular audits and updates into their workflow.

The constitutional orders require not only formal rules but social trust: each participant's confidence that others will comply, that the system functions as designed, and that the information it provides is reliable. Thankfully, LLM agents tend to follow guidance. Correction via audits are also effective in promoting aligned behavior of agents that comprise a team.

Part of trust concerns how an actor responds to its own myopia with respect to other agents in a process. Agents act on the basis of knowledge proxies they cannot independently verify in full. If documentation cannot be trusted, the entire knowledge infrastructure degrades. The living document policy is a trust-sustaining institution as it promotes trustworthiness of agent documentation via a constitutional commitment. This allows agents to trust the knowledge and knowledge proxies made available to them via agent documentation.

Documentation drift is a trust violation, not merely a documentation error. It misrepresents the system's state to agents who rely on it. Conflict logs document instances where documentation has drifted from reality: agent files referencing non-existent directories, outdated workflow steps, inconsistent cross-references. Failure to consistently correct such shortcomings results in an erosion of the shared interpretive frame.

Ultimately, the corrective for a consistent lack of alignment must be the failure of a project. Failing projects tend to be removed from competitive systems. Those involved in such failure bear the cost of the failure.

The constitutional structure described in this chapter governs a single team. Vallier (2026) develops the complementary macro-level analysis: when lineages of autonomous agents compete across an ecosystem, governance regimes themselves face evolutionary selection, and alignment becomes a constitutional problem rather than an engineering problem. The present chapter supplies the micro-organizational design, which includes authority hierarchies, invariant cores, architectural enforcement, documentation-as-trust, that Vallier's framework treats as given within each lineage. What happens when multiple teams independently develop similar constitutional patterns is the subject of Part IV.
