Chapter 1: The Coding Agent: What It Is and What It Isn’t

Part I: Foundations

1. Defining the Coding Agent

A single coding agent can read files, execute commands, and modify source code. Its productive capacity, however impressive, is bounded by what it can hold in attention at any given moment. A team of agents can distribute labor, specialize, and sustain development at scales no single agent could manage. But coordination is not automatic. How do you get a team of agents to coordinate efficiently toward the completion of a shared goal?

The question is practical before it is theoretical. Agent teams face characteristic problems that emerge from the conditions under which they operate: context limitations that erode interpretive accuracy as tasks grow complex, documentation that drifts from the reality it describes, inconsistencies that propagate across agents with overlapping domains, and authority conflicts when multiple agents hold valid but incompatible directives. Left unaddressed, these problems degrade performance regardless of the underlying model's capability. A team whose infrastructure is disorganized, contradictory, or stale will produce poor results from a powerful model.

This book develops a solution to that coordination problem. It draws on traditions in economics and institutional theory that have addressed analogous challenges in human organizations: Hayek's analysis of distributed knowledge and the communication systems that transmit it, Lachmann's theory of heterogeneous capital, the theory of the firm from Coase and from Foss and Klein, Menger's account of how institutions emerge from repeated interaction, constitutional political economy from Buchanan, and Nonaka's theory of organizational knowledge creation. These traditions provide the theoretical architecture for designing agent teams that maintain themselves, detect their own inconsistencies, and evolve in response to the projects they serve. The approach is grounded in theory, tested against the working systems described in these pages, and organized around a thesis that begins with the question of what, precisely, a coding agent is.

To start, we must identify the relationship between a reasoning system and the environment upon which it acts. Hayek (1952) devoted The Sensory Order to a version of this question applied to human cognition. His answer was that the mind cannot be identified with the brain. The mind, for Hayek, is the brain's organized relationship to the environment: the accumulated connectivity that selects which stimuli reach consciousness and how those stimuli are classified. He describes the organization of the nervous system as having a "highly purposive character" (§4.5), emphasizing that it actively shapes perception rather than passively relaying it. A large language model stands in a comparable position with respect to its project environment. The model's reasoning capability, however impressive, is only one component. The organized relationship between that capability and the environment determines what productive work actually gets done.

That organized relationship is the coding agent: a model attached to infrastructure, where the usefulness of the attachment depends on the infrastructure's organization toward a coherent set of goals. An agent, and even more so a well-structured agent team, is an organization. As an organization, it engages in interpretation, reorganization, and action in ongoing exchange with its environment, relying on order-generating mechanisms that iteratively transform agents, teams, and the environments they work within.

1.1 The Elementary Components

A large language model, itself, is empowered with pattern recognition, language generation, and chain-of-thought inference (Wei et al. 2022) across domains, capacities that rest on the transformer architecture (Vaswani et al. 2017) whose self-attention mechanism allows the model to weight relationships across its entire input sequence. These are substantial capabilities by themselves. However, in order for a project to be coherently managed by these agents, that project requires feedback loops that enable self-correcting behavior and self-guided development (i.e., with little to no external oversight). The user can converse with the model. It can answer questions, draft text, reason about code. While amazing by any standard used to judge AI prior to 2022, the LLM's capabilities are constrained as long as it does not have access to a terminal to execute commands, modify files, or search the web. Tool access provides environmental traction: a command line, a file system, code execution, web search. When combined with a framework that iteratively guides agent activity until a particular goal has been achieved, we consider that model-tool bundle to be an agent, a pattern that Yao et al. (2023) formalized as the interleaving of reasoning traces with tool-mediated actions.

The distinction between a conversational model and an agent with tool access may seem merely technical. It is, in fact, the point at which reasoning can translate to productive activity. Tool access enables interpretation and execution. The model reads a file, classifies its content, and situates it within the project. The model then acts: modifying source code, updating documentation, resolving contradictions, altering the environment that it or other agents will encounter next. Each round of reading and acting changes the conditions under which the next round occurs, initiating the dynamic conversation between agent and environment.

1.2 Agent ≠ Model

The model is a component of the agent, not the agent itself. An agent can be improved without upgrading its model. Likewise, a poorly structured agent may receive little benefit from a model upgrade. Two agents that use the same model and that are given the same task, may produce qualitatively different results if one is configured with only an instructions file and the other with a team of specialized agents. The model in both cases is identical. What differs is the organized environment within which the model reasons and acts.

The model does not originate this infrastructure from nothing. The initial organizational design, which includes which conventions to adopt, how to structure the authority hierarchy, what division of labor serves the project's goals, requires the entrepreneurial judgment that Foss and Klein (2012) identify as the defining function of the entrepreneur. In our case, that is the developer who creates the agent team.

The developer can allocate judgment to an agent. For example, I highly recommend telling a generalist agent that you wish to create an agent for a specific task or to serve as expert over a particular domain. Provide an outline of the task and documentation of relevant procedures and protocols that the agent should use. Then advise the generalist agent to follow best practices when designing and formatting the agent. In doing this, you will have allocated part of your entrepreneurial judgment to the agent for this particular task.

As sources of truth identified and developed, which includes abstract templates, shared procedures, naming conventions, and externally managed documentation, the agent will be increasingly effective in creating and maintaining the infrastructure files themselves. An agent instantiating a specialist from a template or propagating governance updates across repositories exercises productive knowledge within a structure that entrepreneurial judgment established. As you will see later, we can simply ask an orchestrating agent to create agent of type x, with the requisite documentation and outline of structure identified, and to clearly define the agent's role with respect to other agents on the team who occupy adjacent roles. The capacity to generate organizational infrastructure is itself a product of the organizational capital this book describes.

The feedback loop extends further than a developer reading agent output and revising conventions. We will use a daily pipeline that includes remediation stages where agents modify other agents' specifications in response to investigation findings. This can include workflows where an agent identifies a need for developing new specialists. Other parts of the workflow automatically analyze agent drift and incongruencies that develop across various teams for agents that are tasked with the same function. The infrastructure must maintain its form while still allowing for legitimate development of new functions. The map must simultaneously constrain and respond to the activity of agent teams.

The infrastructure that organized the agent's initial work is, in turn, reorganized by agents operating within it, not because agents originated the design, but because the design furnished them with the sources of truth and procedural knowledge to extend it. The initial design is an act of entrepreneurial judgment. The ongoing reorganization is a distributed process in which agents participate as productive actors within the structure that judgment created. This dynamic, infrastructure organizing the agent's work, the agent's work reshaping the infrastructure, is what Chapters 12 and 18 develop.

1.3 The Infrastructure Files

The infrastructure to which the model is attached takes concrete form as documentation files. A project-level instructions file (copilot-instructions.md) is, for the human reading it, a TLDR readme file that provides a succinct overview of the agent infrastructure for a repository. More deeply, it provides constitutional scaffolding for the team of agents in relationship to the project that they produce and govern, specifying conventions, priorities, and an authority hierarchy. Agent-specific files (with the .agent.md extension) provide role-specific domain knowledge, tool access boundaries, and workflow procedures. These rules and roles are further elaborated in the repository's agent documentation

These files do not program the model in a deterministic sense. They exploit what Brown et al. (2020) identified as in-context learning: the model adjusts its behavior to instructions provided in the input without modification of its parameters. The rules they provide are intended to filter the behavior of the agent. The infrastructure determines what the model can perceive and attend to, what tools it can invoke, and toward what ends its reasoning is directed, without dictating the specific path that reasoning takes. Hayek (1952, §5.63) describes the nervous system's connectivity in similar terms: it is not a program that dictates responses but an organized structure whose prior connections select which stimuli reach consciousness and how they are classified. Only when the agent is given a well defined directive is it acting rationally, with that rationality being filtered by the rules and capabilities delineated in the agent documentation.

The process of constructing these files is a knowledge-creating activity in its own right. Nonaka (1994) identifies externalization as the mode of knowledge conversion in which tacit understanding is rendered into explicit, codified form. Creating an agent file is an act of externalization: the developer converts an accumulated insight, which may have been previously unarticulated, concerning what the agent should attend to, how it should reason, and what it should avoid into a specification that can be read, shared, and revised. The resulting file also establishes what Nonaka and Konno call ba (Nonaka and Konno 1998), a shared context for knowledge creation, since it is the space within which the model's cognition takes shape. Beliefs are externalized into persistent form that shapes future action.

Infrastructure files are not fixed once written. They are sites of ongoing reorganization, subject to the same back-and-forth between use and revision that characterizes any productive system. The team's governance architecture includes agents whose function is to propagate changes when the project's reality shifts, to detect when documentation has drifted from what it describes, and to enforce the principle that every assertion in an agent file must remain current. The infrastructure is both the medium of the agent's cognitive activity and a product of the team's organizational activity: simultaneously the context for interpretation and the site of reorganization.

1.4 The Project as Goal-Bundle

The agent team is oriented toward the promotion of a project. A project is a coherent set of goals toward which infrastructure is organized. The agent's productive capacity depends on alignment between the infrastructure and the project's specific purposes. Documentation that describes a different project's conventions, or that is internally inconsistent, or that lacks the domain knowledge the current project requires, degrades the agent's performance even when the model itself is highly capable.

Foss and Klein (2012) develop an analogous claim for the firm. The firm's hierarchy, routines, and communication channels are organized around the entrepreneur's assessment of how to deploy heterogeneous resources toward a valued output. Without that entrepreneurial judgment, the organizational structure is inert. Expression of judgment by the entrepreneur animates the firm. The firm distributes and structures capabilities in a manner that efficaciously manifests this expression of judgment.

The project shapes the activity of the team. The parallel is both organizational and motivational, though the motivational dimension requires careful articulation. The agent does not choose goals in the way that an entrepreneur exercises creative judgment under uncertainty. Goals are furnished by the principal through the infrastructure. But the agent can be allocated judgment about how to pursue those goals, and when it is, it ranks outcomes, evaluates means, and selects among alternatives. The infrastructure thus does more than channel capability toward ends. It defines the domain within which the agent exercises rational choice. And just as the entrepreneur revises the firm's strategy in light of market feedback, the agent team's engagement with the project may surface information that refines the developer's understanding of the goals themselves.

1.5 Allocated Judgment and Agent Rationality

The previous sections have drawn parallels between agent teams and economic organizations while noting that goals are furnished by the principal. This framing, though accurate about goal-origination, understates what happens once the agent begins to act. Agent operation, once the mandate is received, takes one of two forms. An agent can be given a determinate procedure: "verify that every citation key in the chapter resolves to an entry in bibliography.bib." Here the agent executes. The means are specified along with the end. But an agent can also be allocated judgment within a given domain: "revise this chapter so that the theoretical framework is more clearly connected to the architectural claims." Here the agent must evaluate multiple possible revisions, rank them against criteria that appear in or that were only implicit in the model's training, and select the approach that best satisfies the composite of explicit instructions and implicit quality standards. The second agent is choosing means to pursue ends. This is rational action in the sense Mises (1949) defined it: purposeful behavior directed toward the attainment of definite ends through chosen means (Kirzner 1962).

The agent's preference ordering, if we may call it that, is not a simple scalar function but a composite of its trained parameters (model weights that embody implicit dispositions over output quality, coherence, and relevance), and its system instructions (the explicit constraints and goals that shape its evaluation of outcomes). Preferences interact with environmental context. For example, preference may be influenced by or filtered through agent knowledge (the project state that determines which outcomes are feasible and which means are available). These composites determine the probability distributions over tokens, and therefore over choices, that shape the agent’s decision-making.

The mechanism differs from human deliberation. There is no phenomenological experience of weighing alternatives, no felt sense of preference. But the function is the same: ranking outcomes and selecting means. The agent's response to a revision task is a structured selection, not a random draw from the space of possible revisions, shaped by what amounts to a preference function: one shaped jointly by the model's training and the infrastructure's constraints.

This nuance has consequences for the relationship between the thesis and the theoretical traditions on which it draws. The parallel with Foss and Klein (2012) is not merely organizational but motivational: the agent allocated judgment within its domain exercises something comparable to the entrepreneurial judgment Foss and Klein describe. It is constrained by the same kind of organizational structure that channels human judgment in the firm. The parallel with Hayek (1945) deepens this insight: each agent is not merely a "man on the spot" who holds local knowledge but one who exercises judgment about how to act on that knowledge, and the quality of that judgment depends on the interpretive frame (the infrastructure) within which it operates. Computational models of ecologically rational agents make the stakes of that dependence visible: the interpretive frame is itself subject to selection pressure (Caton 2017). Agents whose heuristics generate adaptive responses to the environment persist. Agents whose frames misread it are removed. An agent team that encodes a poor interpretive infrastructure is not merely less efficient — it is selecting against itself.

The infrastructure files described in §1.3 do more than channel reasoning. They are partly constitutive of the agent's preference ordering. An agent file that says "prioritize theoretical accuracy over coverage" shapes how the model ranks outcomes in every subsequent evaluation. It functions as a component of the agent's composite utility function rather than a mere instruction. The infrastructure shapes what the agent treats as desirable. It is not separate from the agent's rationality but woven into it. Second, these implications extend to the ongoing reshaping of the agent's preference ordering as infrastructure evolves. When the agent-updater (Chapter 12) revises an agent file, the change does more than update instructions. It reshapes the preference ordering through which the agent will exercise judgment on its next invocation. Reorganization operates on the agent's rational apparatus itself.

That account of rationality bears on a challenge that Kirzner (1979) directed at Hayek: Hayek's agents, he argued, behave as "Robbinsean maximizers": agents that optimize over a fully specified set of known alternatives, executing procedures rather than exercising judgment in the face of uncertainty. Computationally, however, the distinction is less sharp than Kirzner supposed (Caton 2025). Any action selected by an agent is optimal in the limited sense that it was the best response according to the policy that governed the selection. What matters is whether that policy is rich enough to encompass discovery. An agent allocated an open-ended revision task does not enumerate all possible revisions before beginning. It iterates over a solution space whose structure it does not know in advance, and the openness of that domain is the computational analog of Kirznerian alertness. The agent investigator of Chapter 18 makes the same point architecturally: its discovery properties arise not from prior enumeration of possible findings but from the structure of its search procedure.

1.6 The Cognitive Dimension of Infrastructure

Institutions necessarily include a cognitive dimension (Caton and Lopez 2020): they structure cognition as much as they constrain behavior, shaping how agents perceive, categorize, and process information (Hayek 1943a). Agent infrastructure is institutional, and, as such, the rules that comprise the infrastructure form the cognitive environment within which the model reasons (Hayek 1962).

An orchestrator's domain routing table provides a useful illustration. The routing table for the agent team that supported the creation of this book ensures the quality of sourcing, of writing style, of logical chains threading the text, and so forth. When the model encounters a bibliography question, the routing table directs its reasoning toward the bibliography manager's domain knowledge, citation verification procedures, and anti-fabrication safeguards. When it encounters a question about voice consistency, the routing table redirects it toward the voice auditor's samples and calibration standards. The table does not answer either question. What it does is organize the model's attention, structuring how the model processes the question before any reasoning about the answer begins.

These cognitive structures evolve as the team's engagement with the project reveals what they fail to capture. The agent-updater (Chapter 12) revises routing tables and agent specifications when the team's composition or the project's structure changes. The conflict-auditor (Chapter 9) detects when routing relationships have drifted from reality. These maintenance agents are among the mechanisms through which the cognitive infrastructure evolves in response to the team's engagement with the project.

Hayek (1937) showed that plan compatibility requires empirical conditions beyond the logic of choice — and institutions supply those conditions. Institutions provide the shared cognitive structure that promotes mutual interpretation of action among interacting actors, a structure that Denzau and North (1994) analyze in detail for human cognition within institutional contexts. Agent documentation is a domain where this cognitive-institutional nexus can be observed directly: the model's reasoning is partly shaped by the documentation it reads.

Nonaka (1994) provides a complementary vocabulary for the routing table: combination, which is the systematic reconfiguration of explicit knowledge sources into a new explicit structure that governs the model's attention. The conventions and vocabulary embedded in shared instructions files provide shared rules for governing action and interaction, narrowing the range of interpretation and strategy in a manner that promotes coherent, mutually constructive behavior as agents pursue their tasks. Nonaka's insight is that organizational knowledge creation is not a matter of processing information but of converting knowledge between tacit and explicit forms that are either personal or shared. The agent infrastructure is a site where that conversion is architecturally visible. Shared documentation transforms personal knowledge into a shared locus of coordination.

1.7 Documentation as Productive Investment

From the perspective of a user accustomed to ad hoc prompting, writing documentation and agent files looks like overhead. The files sit in a directory. They do not do anything visible until the model reads them. The payoff is not immediate.

From the perspective developed here, documentation is the primary productive activity. Every agent file is an investment in the cognitive organization of the team: in Nonaka's (1994) terms, an act of externalization that converts tacit understanding into explicit organizational capital, and one that must be renewed as the project evolves. Every convention is a constraint that reduces the search space within which the model reasons. Every authority hierarchy prevents the model from acting on domains it is not equipped to address. The largest productivity gains in practice come not from model upgrades but from organizational improvements: better agent files, clearer conventions, more carefully targeted domain knowledge. Quality infrastructure improves the predictability and the efficiency with which a model generates an outcome. Model improvements will reinforce the efficiency promoted by quality infrastructure.

The investment, moreover, is not a one-time expenditure. Because the agent's actions change the environment and the documentation's correspondence to that environment degrades over time (what Chapter 21 calls documentation drift) ongoing re-externalization is necessary. If the agent documentation is capital, this capital requires continual reinvestment in order to maintain its quality (for similar, see Harwick 2022). Further, this capital exists within a nexus of overlapping and still-developing plans that are an inherent part of project development (Kirzner 1966).

1.8 Mind and Social Order

I should be clear about the limits of the parallel between the human mind and human society as compared to the LLM and agent teams. LLMs do not have subjective experience. They do not perceive in the phenomenological sense that Hayek's theory of mind addresses. The claim is that the architectural relationship between model and environment, the way infrastructure organizes reasoning capability toward productive ends, bears comparison to the relationship Hayek describes between the individual mind and the social order in which it participates. The parallel is organizational and functional rather than ontological. The model does not experience deliberation as a human mind does, but when allocated judgment it exercises means-ends rationality: ranking outcomes, evaluating means, choosing among alternatives. This functional parallel is the subject of §1.5 above. The phenomenological limitation matters, but it does not negate the functional parallel, and Chapter 22 returns to both dimensions in the context of the broader question of what agent teams are and are not as complex systems.

1.9 Continual Evolution

The scaffolding is only the beginning of the organizational development, not its end. Subsequent chapters detail how a team's engagement with its project generates discoveries and corrections that reshape the team's infrastructure: an investigative agent identifies patterns the original design did not anticipate, a consistency-auditing agent detects contradictions between documentation and reality (Chapter 9), and a maintenance agent propagates revisions to maintain consistency. The dynamic, fully told, illustrates not just the static productivity claim but the ongoing conversation. Infrastructure organizes the agent's work, and the agent's work informs the infrastructure's reorganization.

Value arises from their continuous interaction: what Nonaka calls the knowledge spiral. This continuous interaction is what makes the agent team a knowledge-creating organization rather than a static tool. The knowledge spiral is the process through which the agent team's interpretive engagement with the environment, mediated by infrastructure, generates organizational knowledge that transforms both the team and its relationship to the project.

1.10 Infrastructure as Capital

If infrastructure is the condition for productive action, then infrastructure is capital in the economic sense. Lachmann (1956), developing a line of argument initiated by Menger (1871) and extended by Hayek (1941), insists that capital is not a homogeneous fund. It is a heterogeneous collection of goods whose productive significance derives from the plan within which they are embedded. A piece of equipment is capital because of the role it plays in a plan of production, not because of its physical properties. The contents of these files embody technologies that shape agent capabilities.

Agent documentation files are capital because of the role they play in the organized system through which the model's reasoning capability is made productive. Plan-dependence carries a dynamic implication that Lachmann himself emphasized. When new information causes the plan to be revised, the capital structure must be reorganized accordingly. Agent documentation is susceptible to the same dynamic. As the team's environmental engagement generates new information, which includes an unanticipated pattern, a drifted cross-reference, a shifted directory structure, the plan may be revised and the documentation reorganized to reflect it. Chapter 3 develops this claim fully, working out the implications of heterogeneity, complementarity, and plan-dependence for the agent team's documentation structure.

2. Preview of the Argument

This chapter has defined the coding agent as a model attached to infrastructure, an organization whose productive capacity depends on the alignment between its documentation structure and the project's goals. The developer is the entrepreneur whose judgment initiates this design: choosing conventions, structuring authority, establishing division of labor. But entrepreneurial judgment need not remain with the developer alone. Agents allocated judgment can create and maintain other agents from templates and accumulated sources of truth. As infrastructure develops, the capacity to generate further organizational capital becomes a product of the organizational capital already in place, a self-reinforcing dynamic that the knowledge spiral sustains.

The agent's rationality is not separable from the infrastructure within which it operates. Documentation files carry a cognitive dimension. They constitute the interpretive environment through which the model reasons, classifies, and coordinates with other agents. Infrastructure so understood is capital in the economic sense: heterogeneous, plan-dependent, and subject to reorganization as engagement with the project generates new knowledge. When allocated judgment, the agent exercises means-ends rationality within this domain: ranking outcomes, evaluating means, selecting among alternatives. When the agent-updater revises an agent file, it reshapes the preference ordering within which that agent will exercise judgment on its next invocation. Reorganization operates on the rational apparatus itself.

The chapters that follow develop this thesis along three axes. Parts I through III establish how the agent's interpretive and active capacities are structured: the context window constrains interpretation (Chapter 2), documentation capital embodies the technology of action (Chapter 3), team formation and constitutional rules organize both (Chapters 3–4), and individual agents instantiate specialized interpretive and active roles (Chapters 5–15). Parts IV and V develop the reorganizational dimension: cross-repository boundary governance is introduced (Chapter 16), abstract templates evolve through institutional emergence (Chapter 17), the investigator discovers what the system does not yet know about itself (Chapter 18), and the daily pipeline drives the dynamic conversation between the system's current state and its documented self-understanding (Chapters 19–21). Part VI synthesizes: the order-generating mechanisms that transform agents, teams, and their relationship to the environment are the subject of the final chapter (Chapter 22).

The chapter-by-chapter development proceeds as follows. Chapter 2 takes up the constraint that shapes the entire architecture, the context window, and argues that each agent in a multi-agent team operates as what Hayek (1945) called the "man on the spot," dependent on knowledge proxies and an interpretive frame rather than centralized knowledge. Chapter 3 develops the capital-theoretic interpretation of agent documentation, drawing on Lachmann's (1956) theory of heterogeneous capital to argue that technology is embedded in agent files and that the structure of that capital is intelligible only through the plan that organizes it.

Chapters 3 and 4 analyze the agent team as an organizational form. Chapter 3 draws on Coase to explain why multi-agent teams exist and traces the division of labor within the team, analyzing the bifurcation between functional agents and domain agents through Smith's (1776) framework of specialization, Hayek's (1945) framework of distributed knowledge, and D. Klein's (2012) concept of knowledge concatenation. Chapter 4 examines the constitutional constraints that govern the team, drawing on Buchanan and Tullock.

Chapters 5 through 15 examine the individual agents that make up a working agent team. Chapters 5 and 6 describe the coordinative agents: the orchestrator's workflow routing and the navigator's structural cartography. Chapter 7 analyzes the security agent's constitutional enforcement mechanism, including its veto power and trigger-based activation. Chapters 8 through 10 cover the quality-assurance triad: the code-hygiene agent's typed audit rules, the conflict-auditor's ten-category detection system, and the conflict-resolution agent's disposition framework. Chapter 11 examines the adversarial agent, which extends the quality-assurance architecture from reactive detection to proactive presupposition critique. Chapters 12 and 13 examine the maintenance agents: the agent-updater's change propagation and the agent-refactorer's structural compliance enforcement. Chapter 14 describes the cleanup agent's four-gate safety architecture for controlled deletion. Chapter 15 surveys eight of the nine domain agents, which includes book-content expert, expert-critic, argument-weaver, voice-style, HTML-document builder, Graphviz expert, LaTeX specialist, and bibliography manager, as a coordinated production chain. The ninth, the dynamically instantiated chapter expert, is covered in Chapter 5.

Chapter 16 examines cross-repository governance: when an agent team's actions propagate beyond its own repository, the repo liaison governs information flow across boundaries without acquiring productive authority or bypassing adjacent teams' constitutional rules. Chapters 17 and 18 trace the emergence of organizational patterns that transcend individual projects. Chapter 17 analyzes how abstract agent types (templates that capture the invariant logic of a function while leaving project-specific details to be instantiated) emerge through a process analogous to Menger's (1883) theory of institutional emergence. Chapter 18 examines the agent investigator as a Kirznerian (1973) entrepreneur, an agent that discovers previously unnoticed discrepancies and routes them to the appropriate agent for remediation. Chapters 19 through 21 describe the automated governance pipeline that maintains the ecosystem: the daily enrichment pipeline as a discovery procedure in Hayek's (1968) sense, the investigation-remediation cycle as a feedback system, and the problem of drift and contradiction in institutional memory.

Chapter 22 synthesizes the argument. The agent team is a designed system that exhibits properties characteristic of spontaneous orders: emergent patterns, adaptive responses to changing conditions, organizational structures that evolve through use rather than decree. The tension between design and emergence is a feature to be explained, and the explanation draws on the full range of theoretical traditions the book has developed. I argue that these traditions provide explanatory power for understanding why agent teams are designed the way they are, why certain organizational patterns emerge and persist, and why the gap between model capability and productive output cannot be closed by model intelligence alone but requires the infrastructure through which that intelligence is organized. The agent team is a domain in which these traditions can be tested against observable architectural patterns.

But the agent's dependence on infrastructure has a deeper source than anything discussed so far. The model's reasoning capacity, however powerful, is constrained by what it can hold in mind at once: a constraint on the agent's interpretive capacity that shapes the entire architecture. Chapter 2 takes up the context window as the communication problem that necessitates the entire architecture.
