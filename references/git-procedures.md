# Git Operating Procedures — AgentTeamsModule

**Canonical reference for all git operations performed by agents or via operator command in this repository.**

Route git operations through `@git-operations`. This document is the authority for command selection, ordering, and recovery strategy.

---

## Category A — Commit and Push

### A.1 Pre-Commit Checklist

Before staging any changes, verify:

1. Tests pass: `python -m pytest tests/ -q`
2. No absolute filesystem paths in staged content (OPSEC — see `@security` trigger list)
3. `docs/` rebuilt if `docs_src/` was modified: `mkdocs build --clean`
4. No backup files (`.bak`, `.bak[0-9]`) or `__pycache__` staged

If any check fails, resolve before staging.

### A.2 Staging

Stage only relevant files — never use `git add .` blindly:

```bash
# Stage specific files
git add <file1> <file2>

# Stage a directory subtree
git add docs/

# Verify what is staged before committing
git status --short
git diff --cached --stat
```

Files to explicitly **exclude** from staging:
- `tmp/__pycache__/`
- `*.bak`, `*.bak[0-9]*`
- `.DS_Store`, `Thumbs.db`
- `agentteams.egg-info/` (regenerated by pip)
- Any file containing absolute `/Users/` or `/home/` paths

### A.3 Commit Message Format

Follow Conventional Commits (`<type>(<scope>): <summary>`):

| Type | Use for |
|---|---|
| `feat` | New feature or capability |
| `fix` | Bug fix |
| `docs` | Documentation changes only |
| `refactor` | Code restructuring without behavior change |
| `test` | Test additions or corrections |
| `chore` | Dependency, CI, config, or housekeeping changes |
| `security` | Security hardening or vulnerability remediation |

Rules:
- Summary line ≤ 72 characters
- Use present tense imperative: "add feature" not "added feature"
- If the change is non-trivial, add a body paragraph explaining WHY, not just WHAT
- Reference issue/PR numbers in the footer: `Closes #42`

Example:
```
docs: add feature inventory page; share summary table via snippet

- Add docs_src/api-reference/feature-inventory.md: clean category-based
  inventory of 125 features organized by capability area
- Extract shared summary table to docs_src/assets/feature-summary-table.md
  so Home and Feature Inventory pages stay in sync from one source
- Remove features-by-date.md: date attributions were inferred, not verified
```

### A.4 Push

```bash
# Push to the configured remote tracking branch
git push

# If no upstream is set yet
git push --set-upstream origin <branch-name>

# Confirm remote advanced
git log --oneline -3
git status  # should show "up to date"
```

If push is rejected due to diverged remote:
- **Do NOT use `git push --force`** — use `--force-with-lease` only after understanding what will be overwritten
- Preferred: pull first, resolve any conflicts, then push (see Category B)
- `git push --force-with-lease` is safe only if you authored all commits being replaced and the remote has no new work from others

## Category B — Integrating Changes from Main

### B.1 Pre-Integration Check

Always fetch and inspect before integrating:

```bash
git fetch origin
git status
git log --oneline HEAD..origin/main    # commits on remote not yet local
git log --oneline origin/main..HEAD    # your local commits not yet on remote
```

If `HEAD..origin/main` is empty, your branch is up to date — no action needed.

### B.2 Fast-Forward Pull (No Local Changes)

When local branch has no new commits and remote has new ones:

```bash
git pull --ff-only
```

If `--ff-only` fails, the branches have diverged — use B.3 or B.4.

### B.3 Rebase then Pull (Linear History)

Preferred for feature branches that haven't been shared:

```bash
git fetch origin
git rebase origin/main
```

If conflicts arise during rebase, see B.5. After a clean rebase:

```bash
git push --force-with-lease    # only for unshared branches
```

### B.4 Three-Way Merge (Diverged Branches, Shared History)

When `main` and the working branch both have new commits and the branch has been shared (pushed):

```bash
git fetch origin
git merge origin/main
```

Git will create a merge commit. If there are no conflicts, the merge completes automatically. If conflicts arise, see B.5.

### B.5 Manual Conflict Resolution Protocol

When `git merge` or `git rebase` reports conflicts:

1. **Identify conflicting files:**
   ```bash
   git status --short   # look for "UU" (unmerged)
   git diff --diff-filter=U   # show only conflicting hunks
   ```

2. **Inspect each conflict:**
   ```bash
   git diff <file>    # shows conflict markers in the file
   ```
   Conflict markers:
   ```
   <<<<<<< HEAD         ← your current branch version
   your content
   =======
   their content
   >>>>>>> origin/main  ← incoming branch version
   ```

3. **Resolve each file** by editing to the intended final state — remove all `<<<<<<<`, `=======`, `>>>>>>>` markers.

4. **Stage resolved files:**
   ```bash
   git add <resolved-file>
   ```

5. **Complete the merge or rebase:**
   ```bash
   # For merge
   git merge --continue

   # For rebase
   git rebase --continue
   ```

6. **Abort if needed** (restores pre-merge state):
   ```bash
   git merge --abort
   # or
   git rebase --abort
   ```

7. **Post-resolution verification:**
   ```bash
   python -m pytest tests/ -q
   mkdocs build --clean   # if docs were affected
   git log --oneline -5   # confirm clean commit history
   ```

### B.6 Post-Integration Verification

After any integration (pull, merge, rebase):

```bash
git log --oneline -5         # verify commit sequence makes sense
git diff HEAD~1 --stat       # sanity-check what changed
python -m pytest tests/ -q   # regression check
```

## Category C — Recovery via git diff and git revert

### C.1 Inspecting Changes with git diff

```bash
# Working tree vs. last commit (unstaged changes)
git diff

# Staged changes vs. last commit (what will be committed)
git diff --cached

# Differences between two commits
git diff <commit-a>..<commit-b>

# Single file across two commits
git diff <commit-a>..<commit-b> -- path/to/file.py

# Summary of what changed (names + line counts only)
git diff --stat <commit-a>..<commit-b>

# Show all changes since the last tagged release
git diff $(git describe --tags --abbrev=0)..HEAD
```

### C.2 Retrieving a File at Any Historical Revision

To read a file as it existed at a past commit **without changing your working tree**:

```bash
git show <commit>:<path/to/file>

# Example: view analyze.py as it was 3 commits ago
git show HEAD~3:agentteams/analyze.py

# Pipe to a pager or file
git show HEAD~3:agentteams/analyze.py > /tmp/analyze_old.py
```

### C.3 Restoring a Single File Without Switching Branches

To restore a specific file to its state at a past commit (overwrites the working-tree copy):

```bash
git checkout <commit> -- path/to/file

# Example: restore a corrupted git-procedures.md from the previous commit
git checkout HEAD~1 -- references/git-procedures.md

# Then stage and commit the restoration
git add references/git-procedures.md
git commit -m "fix: restore git-procedures.md from HEAD~1 (file corrupted)"
```

⚠️ This overwrites the file immediately. Use `git show` first to inspect the version you are restoring.

### C.4 Reverting a Specific Commit (Non-Destructive)

`git revert` creates a **new commit** that undoes a prior commit. It never rewrites history, making it safe on shared branches:

```bash
# Revert a single commit (opens editor for commit message)
git revert <commit-hash>

# Revert without opening an editor
git revert --no-edit <commit-hash>

# Stage revert but don't commit immediately (review before committing)
git revert --no-commit <commit-hash>
git diff --cached   # verify the staged revert is correct
git commit -m "revert: undo <describe what is being reverted>"
```

### C.5 Batch Revert of a Commit Range

To undo a range of commits in a single operation:

```bash
# Revert commits oldest-first without auto-committing each
git revert --no-commit <oldest-commit>..<newest-commit>

# Verify the net diff
git diff --cached --stat

# Commit the combined revert
git commit -m "revert: undo commits <range> — <reason>"
```

### C.6 Stashing Work in Progress

Before a risky operation (merge, rebase, checkout), preserve uncommitted work:

```bash
# Stash all tracked changes (and optionally untracked)
git stash push -m "wip: description of what is being stashed"
git stash push --include-untracked -m "wip: including new files"

# List stashes
git stash list

# Restore the most recent stash
git stash pop

# Restore a specific stash without removing it
git stash apply stash@{2}

# Drop a stash after confirming it is no longer needed
git stash drop stash@{0}
```

### C.7 Locating a Regression Commit (Appendix)

`git bisect` is a binary-search tool for locating the exact commit that introduced a regression. It is a diagnostic tool, not a routine operation:

```bash
# Start a bisect session
git bisect start
git bisect bad                  # current HEAD is broken
git bisect good <known-good-commit>   # last known working commit

# Git checks out a midpoint commit; run tests manually, then:
git bisect good    # if tests pass
git bisect bad     # if tests fail

# Repeat until git identifies the first bad commit.
# Always end the session:
git bisect reset
```

## Quick Reference: Decision Tree

```
Need to push new work?
  → A.1 pre-commit checklist → A.2 stage → A.3 commit message → A.4 push

Need to get changes from main?
  → B.1 fetch + inspect
    → No divergence → B.2 fast-forward pull
    → Your commits only, not shared → B.3 rebase
    → Shared branch with divergence → B.4 three-way merge
      → Conflicts? → B.5 manual resolution

Need to recover something?
  → Just inspect a past version → C.2 git show
  → Restore a single file → C.3 git checkout <commit> -- <path>
  → Undo a commit safely → C.4 git revert
  → Undo a range of commits → C.5 batch revert
  → Preserve WIP before risky op → C.6 git stash
  → Find which commit broke something → C.7 git bisect (appendix)
```
