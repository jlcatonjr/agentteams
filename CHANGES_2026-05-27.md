## Change 1 ŌĆö Recall-First Clause in Audit/Validation Templates

### Problem

Only 4 of 37 emitted agents in a typical copilot-vscode team (`orchestrator`, `navigator`, `adversarial`, `work-summarizer`) call the memory-index. Specialist agents that mid-task want to ask "has this been adjudicated / validated / researched before?" either skip the index (burning context on grep) or have to delegate to `@navigator` (paying a cold-start handoff). The recall-first norm is strong at the top of the team and weak across the long tail of specialists.

### Decision

Channel-through-navigator was rejected because:

- A `--query-index` call is ~1s; an agent handoff is much more expensive (cold start + context re-derivation).
- Specialists need recall *inside* their reasoning loop (e.g., adversarial's T/C presupposition check already inlines the call ŌĆö that's the precedent).
- Centralizing only fits "where is X?" user-driven location, not workflow-internal recall.

Solution: inject a domain-scoped, fenced `memory_index_consultation` clause into the six audit/validation/research templates whose workflows benefit most.

### Templates Touched

Each template gained a `<!-- AGENTTEAMS:BEGIN memory_index_consultation v=1 --> ... END -->` block tailored to that agent's question shape:

- **`conflict-auditor.template.md`** ŌĆö recall before adjudicating `HC` / `SR` / ambiguous `CC` conflicts; "history layer, not authoritative" caveat with `SR` (Stale Reference) fallback rule.
- **`conflict-resolution.template.md`** ŌĆö recall for prior `ACCEPT`/`REJECT`/`REVISE` precedent before applying decision rules.
- **`quality-auditor.template.md`** ŌĆö recall for recurrent defect patterns ("have we flagged this LLM tell before?") before opening a new finding.
- **`technical-validator.template.md`** ŌĆö recall with **lexical** strategy first for precise symbol / path queries, vector fallback for thematic questions.
- **`retrieval-integrator.template.md`** ŌĆö dogfoods the team's own retrieval layer; recall prior contract revisions but resolve against current code per the Invariant Core.
- **`tool-doc-researcher.template.md`** ŌĆö recall prior tool research before opening Tier 1 external docs; reuse verified fields, re-verify the `docs_url`.

### Fence design

The `memory_index_consultation` sid is **new** to the templates. `agentteams --update --merge` adds it as an additional fenced sub-region inside the existing `content` fence. Existing project files merge cleanly (verified against `collector-management`: all 5 active targets emitted the fence on the first `--update` run; `tool-doc-researcher` is dormant in that project because `UNRESOLVED_TOOL_LIST` is empty).

### Coverage delta

After: 9 of 37 agents recall-first (audit/validation cluster newly enrolled).

## Change 2 ŌĆö `emit.py` shrink-warn fix and dry-run annotation

### Problem

`agentteams --update --merge` against an established team produces shrink-warn notices like:

```
Notice: security.agent.md: fence 'threat_intelligence': lost concrete refs: CVE-2026-0300, CVE-2026-20182, CVE-2026-42208 (+2 more)
Notice: references/security-vulnerability-watch.reference.md: fence 'threat_data': lost concrete refs: ...
```

Reading this as a `--dry-run` user, it looks like the update will destroy live CVE data. In fact:

- The `threat_intelligence` and `threat_data` fences are filled by `build_security_placeholders` from live CISA KEV / NVD / OSV feeds at every emit. CISA KEV is a moving top-N window ŌĆö CVEs rotate out by design.
- The canonical CVE history is the cache JSON (`references/security-vulnerability-watch.json`), not the in-fence snapshot.
- `emit.py` already has a `lost_fence_bodies` recovery path (the W22 sidecar mechanism) that writes `.lost.<sid>.md` to the backup dir on real runs. Nothing is ever permanently lost.

So: the warning was technically accurate (the specific CVE IDs *would* disappear from the fence) but semantically misleading (no operator-visible loss; routine feed rotation).

### Code changes

**`agentteams/emit.py:139` ŌĆö new fence-id allowlist:**

```python
# Fences whose body is refreshed each run from an upstream live feed
# (CISA KEV, NVD CVSS, OSV.dev, etc.). Content "loss" in these fences reflects
# normal feed rotation, not user-content deletion ŌĆö suppress the shrink-warn
# heuristic so structural --update runs don't emit alarming false positives.
# The canonical history for these feeds is the cache JSON, not the embedded
# snapshot. Real user content sits in adjacent operator-managed fences.
_LIVE_DATA_FENCES: frozenset[str] = frozenset([
    "threat_intelligence",
    "threat_data",
])
```

**`agentteams/emit.py:_detect_fence_shrink` ŌĆö early-return for live-data fences:**

```python
def _detect_fence_shrink(sid: str, existing_block: str, new_block: str) -> str | None:
    """...
    Live-feed fences (`_LIVE_DATA_FENCES`) are exempt: their bodies are
    refreshed each run from upstream feeds and rotation is expected. The
    sidecar mechanism in `lost_fence_bodies` still preserves the prior body
    on disk if real recovery is ever needed.
    """
    if sid in _LIVE_DATA_FENCES:
        return None
    # ... existing heuristic continues ...
```

**`agentteams/emit.py:911` ŌĆö dry-run notice annotation:**

```python
# Annotate with sidecar-preservation hint so operators don't
# mistake the warning for irreversible data loss.
for notice in mr.shrink_notices:
    annotated = f"{rel_path}: {notice} (prior body will be preserved in a .lost.<sid>.md sidecar in the backup dir on the real run)"
    result.notices.append(annotated)
    if result.dry_run_report is not None:
        result.dry_run_report.notices.append(annotated)
```

### Behavior delta

- `threat_intelligence` / `threat_data` fences: no shrink-warn ever.
- All other fences: shrink-warn still fires; sidecar still written; user-facing message now references the sidecar path explicitly so the operator knows recovery is automatic.
- Real-run notices already included the `ŌĆö recovery: {sidecar}` suffix (existing behavior). The dry-run path didn't ŌĆö that's the gap this change closes.

# Addendum ŌĆö W22 Recall-First Follow-Up (same day, after evaluation)

A 13-query evaluation (`recall-first-index-eval-2026-05-27.py`) against the just-shipped templates revealed only 3/13 queries firing at the templates' Ōēź0.5 threshold. Two structural causes diagnosed: (1) `agentteams.build_team._memory_index_sources` hardcoded the corpus to skip `docs/`, leaving 50+ durable plans invisible to recall; (2) the unified Ōēź0.5 threshold is unreachable for sparse-TF-IDF cosine scores (empirical cap ~0.42 for true top-1 hits; noise floor ~0.12).

After adversarial + conflict audits of the follow-up plan (`recall-first-followup-2026-05-27.plan.md` in the consumer project's tmp/), three coordinated changes landed in this repo.

## Change 3 ŌĆö `memory_index_extra_dirs` brief.json field

### Problem

Consumer projects can't tell agentteams about their own durable-doc directories. Hardcoded sources: `workSummaries/`, `docs_src/` (non-recursive), `references/`, plus a tiny top-level allowlist. Projects that put plans in `docs/` (collector-management has 226 such files) get zero recall coverage on them.

### Code changes

**`agentteams/analyze.py`** ŌĆö copy the new field through `build_manifest`:

```python
"memory_index_extra_dirs": description.get("memory_index_extra_dirs") or [],
```

**`build_team.py` ŌĆö `_memory_index_sources`** extended (+33 lines, before final `return`):

- Reads `manifest["memory_index_extra_dirs"]` (list of strings).
- Each entry: if it contains `*`/`?`/`[` ŌåÆ treated as a project-relative glob expanded by `Path.glob`; otherwise ŌåÆ treated as a directory, recursively scanned for `*.md`.
- Safety: rejects absolute paths; uses `Path.resolve()` + `relative_to(project_root)` to reject `../` traversal; runs each yielded path through `os.path.realpath()` and re-checks `relative_to(project_root)` to defeat symlink escapes.
- Backwards compatibility: absent field ŌåÆ empty list ŌåÆ no change to legacy behavior.

### Tests added

`tests/test_memory_index.py` gains three tests:

- `test_memory_index_sources_picks_up_extra_dirs_directory` ŌĆö recursive directory scan.
- `test_memory_index_sources_picks_up_extra_dirs_glob` ŌĆö glob pattern expansion is literal (non-recursive `docs/*.md` does not pick up `docs/sub/deep.md`).
- `test_memory_index_sources_rejects_path_traversal_and_symlinks` ŌĆö `../outside/`, absolute `/etc/`, and a symlink under `project_root` pointing outside the tree all yield no sources.

All three green. Total memory-index test count: 39 (was 36).

## Change 4 ŌĆö Per-strategy thresholds + lexical-first in 4 templates (`v=1 ŌåÆ v=2`)

### Problem

Templates `conflict-auditor`, `conflict-resolution`, `quality-auditor`, `retrieval-integrator` defaulted to vector strategy with Ōēź0.5 threshold. Evaluation showed vector top-1 caps at ~0.42 for real hits ŌĆö the threshold is unreachable. Templates `technical-validator` and `tool-doc-researcher` already used lexical strategy and empirically fire reliably; that's the right default.

### Template changes

All 4 templates: rewrite the `memory_index_consultation` fence body, bump version `v=1 ŌåÆ v=2`:

- **Strategy default:** lexical-first.
- **Fallback rule:** vector fallback triggers when (a) lexical returns zero hits OR (b) lexical top-1 has no content-word overlap with the query (single-term false-positive guard, per adversarial finding adv-4) OR (c) the question is purely thematic with no concrete handle.
- **Per-strategy thresholds:** lexical reliable Ōēź3.0 / candidate Ōēź1.0; vector reliable Ōēź0.30 / candidate Ōēź0.20; cap acknowledged at ~0.42.
- **Cascade end:** if neither strategy produces a candidate, fall back to grep / filesystem.

### Why bump v=1 ŌåÆ v=2

The fence body changed guidance semantics (strategy default + threshold values), not just wording. The version bump signals real drift to downstream consumers running `agentteams --bridge-check` (per adversarial finding adv-5). Body-only changes within v=1 are reserved for prose-equivalent edits.
