# Investigation: Does the bridge `offline=False` hardcode have a purpose, or is it a refactor artifact?

**Date:** 2026-06-15 · **Scope:** `_run_bridge` (and its siblings `_run_convert`/`_run_interop`) build security placeholders with a hardcoded `offline=False` and then assert `_assert_security_intelligence_fresh`, so `--security-offline` has no effect on those paths. A no-network `agentteams --bridge-from … --bridge-refresh` therefore raises `RuntimeError: security intelligence is stale`.
**Status:** FINAL (audited — adversarial + conflict; see §8).

## 1. TL;DR verdict
- **NOT a refactor artifact (high confidence).** The `offline=False` hardcode predates the refactor: it was added net-new in `b83ea9c "feat(security): add post-production audit hardening"` and applied uniformly to `_run_convert`/`_run_interop`/`_run_bridge`. The refactor (Step B, `4565f9e`) moved it **verbatim**; the bridge path **never** threaded `offline=args.security_offline` in any commit. No refactor dropped offline-honoring — the bridge never had it.
- **Intentional by design — inferred, medium confidence.** The asymmetry (generate honors `--security-offline`; cross-framework runners force a live fetch) is corroborated by a deliberately-built **signed-waiver mechanism** for air-gapped use and by air-gapped docs. No commit message or comment states the rationale explicitly, so "intentional" is an inference (see §4, §7), not a documented fact.
- **What the hardcode actually enforces (corrected):** `--security-offline` does **not** bypass the freshness gate — the *same* gate runs on the generate path too, and offline-with-stale/no-cache is still blocked and still needs a waiver. The only real difference is that `offline=True` permits a **fresh (<24h) cached snapshot**, whereas the cross-framework runners force a **live network fetch**. So the policy is: *destructive cross-framework writes must verify against a live feed, not a cache.*
- **Real weaknesses found:** (W1) the bridge policy is undocumented and the gate's error doesn't mention the escape; (W2) the air-gapped **waiver escape is real but its tooling/docs are broken** — the docs advertise `--create-waiver`/`--verify-waivers` commands that **do not exist** and a CSV schema/HMAC that **don't match the code**, so air-gapped use currently requires hand-crafting a signed CSV; (W3) whether forcing a live fetch (vs allowing a fresh cache) is *too strict* is an open maintainer policy call.
- **Recommendation:** treat this as **documentation + tooling work, not a behavior fix.** Do not add a silent `--security-offline` bypass to the cross-framework paths; if the live-fetch requirement is ever relaxed (W3), keep it auditable. Fix the broken waiver docs/tooling (W2) — that is the genuinely actionable gap. The earlier flaky-test fix (monkeypatch) was correct.

## 2. The question
After the build_team decomposition, an audit flagged that `_run_bridge` ignores `--security-offline` while the generate path honors it. The `references/plans/refactor-remaining.plan.md` (I1) already called this "possibly intentional (bridges may require live intel)" and **deliberately declined to alter bridge security behavior**, choosing to surface it. This report resolves that open question: **artifact-to-fix, or intended behavior?**

## 3. Evidence — git archaeology (decisive; high confidence)
| Fact | Commit | Meaning |
|---|---|---|
| `_run_bridge` created (no offline handling) | `f36d8fc "feat(interop): add convert/interop/bridge workflows…"` | Bridge path origin. |
| `offline=False` + freshness gate **added net-new** to all three cross-framework runners (3 `+offline=False`, 0 `-offline=args.security_offline`) | `b83ea9c "feat(security): add post-production audit hardening"` | Deliberate feature-commit choice, applied to convert/interop/bridge at once. |
| `offline=args.security_offline` introduced — **generate path only** | `e68d422 "feat: integrate live security vulnerability references…"` | The flag was wired into local generation, never into the cross-framework runners. |
| convert/interop/bridge moved to `cli/commands.py` | `4565f9e` (Step B) | **Verbatim move** (independently clearance-verified); the gate block is byte-identical pre/post. |
| generate pipeline moved `app.py`→`generate.py`, offline threading preserved | `96dfafc` / `c0213ea` | The refactor preserved the generate path's offline handling. |

*(Minor: the literal string `offline=False` first appears in `e68d422` as a default parameter value + a test helper — not in any runner. The runner gate is correctly attributed to `b83ea9c`.)*

**Conclusion of §3:** The generate-vs-cross-framework asymmetry is **original design** from two independent pre-refactor feature commits. The refactor changed neither side. **Not an artifact.**

## 4. Evidence — is the asymmetry deliberate? (medium confidence)
The load-bearing evidence for *intent* (as distinct from a consistent copy-paste) is the **independently-built offline machinery**, not the uniform triple:

1. **A signed-waiver escape exists and is real.** `_assert_security_intelligence_fresh` (security_gate.py:198-217): if intel is `stale`, it consults `_latest_security_waiver(action="security-intel-freshness")`; a valid HMAC-signed waiver → consume + pass. This is end-to-end real — `test_security_intelligence_freshness_consumes_signed_waiver` (tests/test_build_team_security_gates.py:308-336) proves a signed waiver lets stale intel pass and exhausts on its use limit. Someone **designed an offline story that routes around the flag** via an auditable waiver.
2. **Docs name air-gapped waivers a first-class path.** `docs_src/security-hardening-guide.md` describes a "Waiver system for offline/air-gapped environments" (§3), `air-gapped` as a waiver reason (§61), and air-gapped CI as a supported scenario (§87).
3. **Corroborating (not load-bearing): the uniform triple.** `offline=False` + the gate appear identically in `_run_convert` (commands.py:50-57), `_run_interop` (134-141), `_run_bridge` (236-243). This proves the asymmetry is **consistent** — but a careful copy-paste of one block across three adjacent runners would look the same, so the triple alone does **not** distinguish "intended" from "consistent accident." It is the waiver mechanism (1) + docs (2) that tip the inference toward deliberate.
4. **The gate is a pure precondition, in all three runners.** `bridge_security` is built with `max_items=1, skip_nvd=True` (a lightweight liveness probe) and passed **only** to the gate; `run_bridge(...)` receives no security argument. The same holds for convert (`convert_security` → gate only; `convert_team` takes no security arg) and interop. So the intel is **not embedded** in any output — the gate asserts "the security feed is live/fresh **before** a destructive cross-framework write," it does not ship intel into the target.

**Design intent (reconstructed, medium confidence):** A destructive write of agent infrastructure into a *target* project must verify against a **live** security feed (not a stale-tolerant cache); air-gapped exceptions are meant to go through a **signed, auditable waiver**, not a silent CLI flag. Local generation, acting on the user's own project, gets the convenience of `--security-offline` (fresh-cache-allowed). The commit message (`b83ea9c`) does not state this; it is inferred.

## 5. Weaknesses (the legitimate residue of the original concern)
- **W1 — Bridge policy undocumented; unhelpful error.** No code comment at the `offline=False` sites; the gate's `RuntimeError` reports status/age/ttl but never mentions `--security-offline` doesn't apply or that a waiver is the escape. (Note: docs §46 *does* name convert/interop in the freshness-check list — only **bridge** is entirely absent — but none explain the offline/flag interaction.)
- **W2 — The air-gapped waiver escape is real but its tooling/docs are broken.** There is **no waiver-minting CLI** — `grep` finds no `--create-waiver`/`--verify-waivers` in code, yet `security-hardening-guide.md` (lines ~122-131) advertises exactly those commands. The docs' waiver CSV header (`issued_at,expires_at,reason,approver,hmac_signature`) and HMAC payload do **not match** the schema the code requires (`timestamp,waiver_id,action_reviewed,expires_at,max_uses,uses,approver,ticket_id,reason_code,conditions_verified,signature`; security_gate.py:69-83, 395-407). So today, air-gapped use requires hand-authoring a correctly-signed CSV by reverse-engineering the code — the "documented escape" is partly fictional. **This is the most actionable defect.**
- **W3 — Live-fetch-vs-fresh-cache strictness is an open policy call.** Because the intel is a pure precondition (not embedded), one could argue cross-framework writes should accept a fresh cache (like generate) rather than force a live fetch. Counter-argument: the hardening model wants destructive external writes verified against a live feed. This is a maintainer judgment, not a code defect.

## 6. Recommendation (advisory — these are maintainer decisions)
1. **Keep the cross-framework live-fetch requirement; do not add a silent `--security-offline` bypass.** *Corrected rationale:* not because offline "bypasses the audit trail" (it does not — the same gate runs and stale/no-cache is still blocked), but because the design choice is *"destructive cross-framework writes verify against a live feed, not a cache."* **Conditional on W3:** if the maintainer decides that policy is too strict, the right relaxation is still an auditable path (e.g., honoring `--security-offline` *and* logging a decision), not a silent flag. The consistency-with-generate counter-argument is legitimate and should be weighed, not dismissed.
2. **Fix the waiver tooling/docs (W2) — highest-value action.** Either build `--create-waiver`/`--verify-waivers` to match the real schema, or correct `security-hardening-guide.md` to document the actual CSV header + HMAC payload. Without this, the air-gapped escape is effectively unusable.
3. **Document the bridge policy + improve the gate error (W1).** Add a comment at the cross-framework `offline=False` sites; have `_assert_security_intelligence_fresh`'s error point to the waiver path and note `--security-offline` does not apply to cross-framework operations.
4. **The flaky-test fix is correct as-is** (monkeypatch the gate, matching sibling bridge tests).

## 7. Confidence & reconciliation
- **High confidence:** *not a refactor artifact* — git history is unambiguous (§3).
- **Medium confidence:** *intentional by design* — inferred from the waiver mechanism + air-gapped docs (§4.1-4.2); no explicit design note exists (`b83ea9c`'s message is generic). The uniform triple is corroboration only, not proof.
- **Reconciliation with the prior position:** `references/plans/refactor-remaining.plan.md` (I1) had already labeled this "possibly intentional (bridges may require live intel)" and **declined to alter it unilaterally**, choosing to surface it. This report does not reverse a "fix it" decision — it **resolves the open question that plan deliberately left to investigation**, and confirms the plan's instinct to escalate rather than patch was correct.
- **Caveat:** the W3 strictness question is a policy call not resolvable from the code; recommendation #1 is conditional on it.

## 8. Audit findings & revisions (adversarial + conflict)
Two independent audits reviewed the draft. The core verdict (not-artifact + intentional) **survived**; the following corrections were folded in:
- **Adversarial (major) — corrected the Claim-4 reasoning.** The draft said threading `--security-offline` "bypasses the signed-waiver audit trail." That is false: `--security-offline` runs the *identical* gate (generate.py:276) and offline-with-stale/no-cache is still blocked, still requiring a waiver. The real distinction (fresh **cache** vs forced **live fetch**) now drives §1, §6.1.
- **Adversarial (major) — qualified the waiver escape (W2).** The escape is test-verified real, but the documented CLI (`--create-waiver`/`--verify-waivers`) doesn't exist and the docs' CSV/HMAC don't match the code. Added W2 and recommendation #2.
- **Conflict (most important) — confidence alignment + evidence ordering.** §1's "intentional" is now hedged to medium confidence to match §7; §4 now leads with the waiver mechanism (the real intent evidence) and demotes the uniform triple to corroboration (a triple proves consistency, not intent).
- **Conflict — honest reconciliation of the prior "product gap" framing** (§2, §7): this resolves an open question the prior plan flagged, not a reversal of a firm fix decision.
- **Conflict — recommendation #1 made conditional on the W3 policy axis** and the consistency-with-generate counter-argument acknowledged rather than dismissed; imperative voice softened to advisory.
- **Minor (adversarial):** tightened W1 (bridge specifically absent from docs; convert/interop partly listed) and noted the `offline=False` string first appears in `e68d422` as a default/test, with the runner gate correctly attributed to `b83ea9c`.
